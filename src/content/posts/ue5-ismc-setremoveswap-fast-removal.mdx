## 何が問題なのか

Instanced Static Mesh Component（ISMC）は、同じメッシュを大量に描画してもドローコールが増えない便利な仕組みです。ところが**インスタンスの削除**には落とし穴があります。デフォルトでは削除後もインスタンスの並び順を維持するため、抜けた穴を埋めるように残りのインスタンス全体をずらす処理が走ります。つまり削除コストはインスタンス数に比例（O(N)）します。

数十個なら気になりませんが、弾丸・破片・パーティクル的な用途で数千〜数万個を高頻度に出し入れすると、この「ずらし処理」だけでフレームが止まります。

## 解決策: SetRemoveSwap()

ISMCには `SetRemoveSwap()` という設定が用意されています。これを呼ぶと、削除時に「穴を埋めるために全体をずらす」代わりに、**削除対象と末尾のインスタンスを入れ替えてから末尾を切り落とす**方式になります。C++の `TArray::RemoveAtSwap()` と同じ発想です。

```cpp
// 順序維持が不要なISMCには生成時に一度呼ぶだけ
InstancedMeshComponent->SetRemoveSwap();
```

## どれくらい速くなるのか

元記事の筆者（NukeTheBees氏）が自作ゲームで、ピーク時2万発超のISMC弾丸を対象に20秒間のベンチマークを取っています（AMD 5800X3D / RTX 5070。この設定の有無だけを変えた比較）。

## トレードオフと使いどころ

代償はひとつだけで、**インスタンスの並び順が保証されなくなる**ことです。

- インスタンスに対応するデータを並行配列（インデックス対応）で持っている場合は、そちらも `RemoveAtSwap()` で同期させる必要があります
- 並び順に意味があるロジック（インデックスでの参照など）がある場合は使えません

逆に言えば、弾丸・デブリ・草の刈り取りのような「順序がどうでもいい大量インスタンス」には入れない理由がほぼありません。公式のISMCチュートリアルではほとんど触れられていない設定なので、高頻度に削除が走るISMCを持っているプロジェクトは一度プロファイルを取って確認する価値があります。

なお、当サイトでの再現ベンチマークは未実施です（数値はすべて元記事の実測値）。
