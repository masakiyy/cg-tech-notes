# 下書き生成ガイドライン

定期実行タスク（または手動起動した Claude セッション）が、`data/inbox.json` の収集候補から
記事の下書きを生成する際のルール。**このガイドラインに従わない下書きは作らないこと。**

## 手順

1. `npm run collect` を実行して候補を更新する
2. `data/inbox.json` を読み、下の選定基準で有望なネタを選ぶ（1回の実行で最大3本）
3. 選んだネタは可能な範囲で一次情報（公式ドキュメント・リリースノート・元投稿）を確認する
4. 下書きを `drafts/<slug>.mdx` に書く（slug は英語ケバブケース）
5. 下書きにしたネタ・明らかなゴミ（宣伝・作品自慢のみ等）は `data/inbox.json` から削除する
6. 実行レポートに「生成した下書き一覧」「スキップ理由」「取得失敗したフィード」を書く

## 選定基準（優先順）

1. **公式の隠れ機能** — ドキュメントには載っているが知られていない機能・設定・ショートカット
2. **非公式・現場の使い方** — 公式が案内していない実用テクニック、ツールの組み合わせ
3. **最先端テック** — 大きめの技術ニュース（新レンダラ、新機能、研究発表）

スキップするもの: 単なる作品紹介、セール・宣伝、根拠の薄い噂、既に書いた内容と重複するもの。

## 執筆ルール

- **自分の言葉で再構成する。** 元記事・投稿の文章をコピー・逐語訳しない
- **出典を必ず frontmatter の `sources` に入れる**（元投稿・公式ドキュメントの両方）
- 効果が比較で示せる内容は **BeforeAfter コンポーネント**を使う（設定値の差分でも可）
- 実際に検証できていない内容は本文に「未検証」と明記する
- 文体: です・ます調。手順・設定値・ショートカットは具体的に書く
- AI臭の排除: 「〜と言えるでしょう」の連発、意味のない箇条書きの乱発、
  「いかがでしたか」等の常套句、大げさな形容詞を使わない

## frontmatter テンプレート

```mdx
---
title: '記事タイトル（機能名・効果を具体的に）'
description: '1〜2文の要約。何ができて何が変わるか。'
pubDate: 2026-07-02
software: [blender]            # blender / houdini / unreal（複数可）
category: official-hidden      # official-hidden / community-tip / news
sources:
  - title: '元情報のタイトル'
    url: 'https://example.com/...'
---

import BeforeAfter from '../../components/BeforeAfter.astro';

本文…
```

## 公開について（重要）

- 下書きは `drafts/` に置くだけにする。**`src/content/posts/` へ直接書かない**
- 公開（承認）は人間が `npm run publish:post -- <slug>` で行う
- git push も人間の承認後のみ
